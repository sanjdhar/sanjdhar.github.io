The Core Problem

While you work for weeks, main moves forward. Your branch and main diverge and you need to reconcile those two lines of history.

main:     A---B---C---D---E   (new work from others)
               \
feature:        F---G---H      (your work)

Merge and rebase are two different strategies to bring these lines back together.

Option 1: Merge git merge

git fetch origin
git merge origin/main

This pulls main's new commits into your branch by creating a merge commit that ties the two histories together:

main:     A---B---C---D---E
               \           \
feature:        F---G---H---M   (M = merge commit)
  • History is preserved exactly as it happened — nothing is rewritten.
  • The merge commit can look noisy, and you get one every time you sync.
  • Conflicts are resolved once, all at once, in the merge commit.
  • Safe — never rewrites existing commits. Ideal if anyone else ever touches your branch or you've already pushed it.

Option 2: Rebase git rebase

git fetch origin
git rebase origin/main

This replays your commits on top of the latest main, as if you'd just started your work today:

main:     A---B---C---D---E
                           \
feature:                    F'---G'---H'   (your commits, rewritten)
  • Linear, clean history — no merge commits.
  • Your commits get new SHAs (F', G', H') — they are rewritten copies.
  • Conflicts are resolved commit by commit as each is replayed — smaller chunks, but potentially more rounds.
  • Because it rewrites history, after rebasing a pushed branch you must use git push --force-with-lease (never plain --force).

A Note on git pull origin main

Be careful: git fetch origin just downloads remote changes without touching your branch. But git pull origin main fetches AND merges in one step.

In Option 1 (Merge)

  • git pull origin main already does the merge — a second git merge origin/main becomes redundant.
  • Use: git fetch origin && git merge origin/main

In Option 2 (Rebase)

  • Do not use git pull origin main first — it creates a merge commit that you then rebase on top of. Messy.
  • Use: git fetch origin && git rebase origin/main

Which Should You Use?

Recommendation for solo feature branches: Since you're the only person working in your feature branch, rebase is usually the better fit. You get a clean, linear history that's easy to review when you finally open the PR — and you don't risk disrupting collaborators (the main danger of rebase). Remember to use git push --force-with-lease after rebasing a pushed branch.

Use merge instead if:

  • Someone else also commits to your branch.
  • Your team policy forbids rewriting pushed history.

Best Practices to Minimize Conflicts

  1. Sync early and often. Don't wait weeks. Pull main into your branch every day or two. Small, frequent reconciliations equal tiny conflicts instead of one giant painful one.
  2. Keep features small. Multi-week branches are conflict magnets. Split work into smaller PRs that merge sooner to drastically reduce divergence.
  3. Commit in small, logical chunks. Smaller commits make rebase conflicts easier to resolve one at a time.
  4. Enable rerere. Git will remember how you resolved a conflict and auto-apply it next time — hugely helpful on long-lived branches where you re-sync repeatedly.
    git config --global rerere.enabled true
  5. Use --force-with-lease, never --force. --force-with-lease refuses to overwrite if someone else pushed changes you haven't seen, protecting you from clobbering work.
  6. Run tests after each sync. A "clean" merge or rebase can still break behavior. Always verify after reconciling.
  7. Communicate on big refactors. If you know main is about to get a large change touching your files, sync right after it lands.
  8. Consider git pull --rebase as your default. Avoids accidental merge commits when updating your local branch.
    git config --global pull.rebase true

TL;DR

  • Merge preserves history (safe, a bit noisy). Rebase rewrites for a clean linear history — great for solo branches, but needs --force-with-lease on push.
  • Don't mix git pull origin main with a manual merge or rebase — use git fetch first.
  • The single biggest conflict-killer isn't merge vs. rebase — it's syncing frequently, keeping branches short-lived, and turning on rerere.
